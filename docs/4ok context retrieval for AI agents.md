# Governed company context retrieval for AI agents

## Executive conclusion

The best fit for your use case is **not** a peer-centric or agent-centric memory system. It is a **hybrid evidence system**: immutable source records and chunks as the ground truth, a **canonical entity graph** for cross-system linking, and **hybrid retrieval** that combines vector search, keyword search, graph expansion, and hard metadata/permission filters **before** answer synthesis. Among the systems reviewed, **Zep** is the closest packaged product to that architecture because it stores raw episodes, derives entities and facts with provenance, supports temporal updates, exposes graph search with metadata/type/time filters, and adds enterprise controls such as RBAC, API logs, BYOK, and BYOC. If you want maximum control in open source, the strongest direction is **Neo4j plus LlamaIndex plus LangGraph**, optionally borrowing ideas from **Graphiti** for temporal invalidation and provenance. ## What the systems are actually optimized for

**The strongest company-context candidates** are **Zep**, **Graphiti OSS**, **Neo4j-based custom graph systems**, and **LlamaIndex with a property-graph layer**. These systems make **documents, episodes, entities, edges, and evidence** primary. Zep and Graphiti are especially well aligned because they ingest raw data as episodes, extract entities and relationships over time, preserve lineage to source episodes, and support type, property, metadata, and temporal filtering at retrieval time. Neo4j gives the most control for explicit enterprise modeling and permission-aware retrieval, while LlamaIndex gives the cleanest open-source building blocks for source-node handling, recursive retrieval, and citation-oriented answer generation. **OpenViking** is more promising for company context than most personal-memory tools because it is not centered on a person or agent. Its core abstraction is a filesystem-like context layer with explicit URIs, summaries, relations, and hierarchical retrieval. ## Recommended architecture

For your use case, the right architecture is a **hybrid evidence lake plus entity graph**, not a pure memory layer. The **ground truth objects** should be raw source records or normalized ŌĆ£episodesŌĆØ from Slack messages, email threads, tickets, docs, CRM notes, and file chunks. Every object should carry immutable source identifiers, timestamps, actor/customer/project metadata, content hashes, and permission metadata. Derived facts and summaries should always point back to those records. This follows the strongest ideas in Zep/Graphiti and avoids the biggest failure mode of personal-memory systems: losing the connection between memory and evidence. ŅłĆciteŅłéturn21view2Ņłéturn24view0Ņłéturn24view1Ņłéturn25view0Ņłü

On top of that, create a **canonical entity layer** for people, customers, accounts, projects, tickets, contracts, documents, teams, and systems, with alias edges back to source-system identifiers. This is where Neo4j-style modeling shines, and it is the main thing Honcho-style peer memory does not solve. If you do not build explicit entity objects and alias mappings, cross-source retrieval will remain brittle because the system will have no stable answer to ŌĆ£Acme,ŌĆØ ŌĆ£Acme Corp,ŌĆØ ŌĆ£customer_id=1234,ŌĆØ ŌĆ£the Q2 renewal,ŌĆØ and ŌĆ£ticket LIN-4821ŌĆØ referring to the same thing. ŅłĆciteŅłéturn36view3Ņłéturn36view0Ņłéturn25view2Ņłéturn23search11Ņłü

Retrieval should be **multi-stage**. First do **global evidence discovery** across all permitted source records using hybrid vector/keyword search plus metadata filters. Then do **entity resolution** and candidate ranking. Then do **graph expansion** around the best candidate entities, projects, customers, or documents. Only then should you assemble an evidence pack and ask the LLM to summarize or answer. This is precisely the opposite of peer-centric memory, which tends to scope first and retrieve second. It also combines the best ideas from GraphRAG local search, ZepŌĆÖs graph search, and LlamaIndexŌĆÖs source-node-aware synthesis. ŅłĆciteŅłéturn33view1Ņłéturn25view1Ņłéturn39search0Ņłéturn40search5Ņłéturn40search9Ņłü

Permissions should be enforced **before candidate selection**, not after answer generation. Neo4jŌĆÖs filtering patterns and ZepŌĆÖs metadata-filtered graph search both point in the right direction here: candidate sets should already be permission-aware when retrieval runs. Controlled reveal then becomes a second pass over the already authorized evidence pack, not a last-minute bandage. This matters for both security and relevance because over-fetch-then-trim is bad for latency, bad for ranking quality, and risky for governance. ŅłĆciteŅłéturn36view2Ņłéturn25view0Ņłéturn21view2Ņłü

Finally, treat **summaries as caches, not truth**. GraphRAG community reports, OpenViking L0/L1 summaries, Mem0 memories, and Letta blocks can all be useful derived views, but they should never be the canonical substrate for a company brain. In a governed system, the things you delete, update, audit, and permission-check must be the raw records and their canonical graph mappings. Summaries can be regenerated; evidence must be traceable. ## MVP now and what to avoid

If you want the **fastest credible MVP**, use **Zep** as the retrieval substrate and build your own ingestion, permission sync, and answer UI around it. It already gives you the most difficult retrieval primitives: temporal graphs, typed entities/edges, provenance to episodes, group/shared graphs, hybrid search, and enterprise deployment options. What you still need to add is your connector layer, your canonical identity/ACL mapping, and your UI/agent workflows. ŅłĆciteŅłéturn24view1Ņłéturn21view1Ņłéturn21view2Ņłéturn23search4Ņłéturn26search0Ņłéturn26search7Ņłü

If you want the **best open-source MVP**, wire together **Neo4j + LlamaIndex + LangGraph**. Use LlamaIndex to parse documents into nodes, keep source-node references, compose retrievers, and produce citation-bearing answers. Use Neo4j for canonical entities, alias resolution, graph traversal, vector/full-text retrieval, and permission-aware candidate filtering. Use LangGraph for orchestration, thread state, and tool-driven agent execution. If temporal invalidation matters from day one, add **Graphiti** concepts or Graphiti itself rather than trying to approximate them later. ŅłĆciteŅłéturn39search0Ņłéturn40search5Ņłéturn40search9Ņłéturn36view0Ņłéturn36view2Ņłéturn28view0Ņłéturn29search21Ņłéturn24view0Ņłü

What I would **avoid** is using **Honcho, Mem0, or Letta as the primary company-wide context layer**. They are not ŌĆ£badŌĆØ systems; they are the wrong first substrate for your stated problem. Honcho is participant-centric. Letta is agent-centric. Mem0 is memory-record-centric, and its OSS graph support was removed in favor of entity-linking boosts. All three can still be useful at the edge for agent continuity or user preference memory, but they should sit **next to** the company brain, not **be** the company brain. I would also avoid making **Microsoft GraphRAG** the primary live context plane. The agent query API for your system should look more like a retrieval service than a memory API. In practice, I would give agents five tools: **discover_evidence**, **resolve_entities**, **expand_entity_context**, **assemble_evidence_pack**, and **answer_from_evidence**. The first tool should search globally with metadata and permission filters. The second should return candidate canonical entities with confidence. The third should expand typed graph neighborhoods and relevant documents. The fourth should materialize the exact snippets, source IDs, timestamps, and line/document links the answer may cite. The fifth should be allowed to summarize only from that pack. That recommendation is an inference from the strengths of graph-first retrieval systems and the weaknesses of agent-memory-first systems reviewed above. ## Open questions and limitations

Two gaps remained after focusing on the highest-confidence sources. First, the **exact current OSS license** was not confirmed from primary sources reviewed here for **Honcho** and **Mem0**, so those license cells should be verified before procurement or adoption. Second, this review focused on the systems you named and adjacent primary-source documentation; it did **not** deeply score other current commercial offerings such as cloud-vendor memory services or newer niche OSS projects. That does not change the architecture recommendation: for a 4ok, the winning shape is still **source-record truth + explicit entity graph + hybrid retrieval + permission-aware evidence packs**, not peer-centric memory alone.
